## 1. [전략이] 가장 큰 문제: 프로젝트들이 따로 논다

### 현재 상황
9개 프로젝트가 각각 독립 앱으로 존재. 사용자는 포탈에서 프로젝트를 골라 들어가야 하고, 프로젝트 간 이동이 없음.

### 문제점
```
현재: 사용자 → 포탈 → [날씨] [지도] [플래너] [코스] [챗봇] ← 각각 별개
이상: 사용자 → 제주패스 → 날씨 보면서 코스 짜고 → 바로 예약
```

- 날씨 지도에서 "맑은 곳"을 봐도 코스 메이커로 넘어갈 수 없음
- AI 플래너가 만든 일정에 날씨 정보가 없음
- 코스 메이커 결과에서 예약으로 연결 안 됨
- 대시보드와 챗봇이 실제 예약 데이터와 연결 안 됨

### 제안: 통합 여행 플로우

**단기 (2주)**: 프로젝트 간 딥링크 연결
```
날씨 지도 "애월 맑음" → [이 지역 코스 만들기] → 코스 메이커 (지역=애월 프리필)
코스 메이커 결과 → [이 코스로 일정 만들기] → 여행 플래너 (코스 데이터 전달)
여행 플래너 결과 → [숙소 예약] [렌터카 예약] → 제주패스 예약 페이지
```

**중기 (1개월)**: 통합 앱으로 합치기
- 9개 독립 앱 → 1개 앱의 탭/섹션으로
- 공통 인증, 공통 상태 관리 (Supabase)
- 사용자 프로필 + 저장된 코스/일정

**장기 (3개월)**: 원스톱 여행 플랫폼
- "제주 3박4일" 입력 → 날씨 반영 일정 + 예약까지 한번에

## 2. [비즈] 매출 연결 고리가 빠져있다

### 현재 기획의 매출 시나리오
기획서에는 연 매출 +8억(플래너), +5억(동적가격), +2억(허브) 등 큰 숫자가 있지만, **현재 구현된 프로젝트에는 결제/예약 기능이 하나도 없음**.

### 가장 빠르게 매출을 만들 수 있는 경로

| 순위 | 경로 | 예상 월매출 | 구현 난이도 |
|------|------|-----------|-----------|
| 1 | **카페패스** — 5잔 묶음 ₩15,000 | 300만~ | 중 (Supabase + 카카오페이) |
| 2 | **날씨→실내 할인** — 비 올 때 실내 시설 할인 딜 | 200만~ | 하 (제휴만 하면) |
| 3 | **코스→렌터카** — AI 코스 결과에서 렌터카 예약 CTA | 500만~ | 하 (링크만 걸면) |
| 4 | **플래너→패키지** — AI 일정의 숙소/액티비티 예약 | 1,000만~ | 상 (PMS 연동) |

### 즉시 할 수 있는 것
- 모든 AI 결과 페이지에 **제주패스 렌터카 예약 CTA** 넣기
- 코스/플래너 결과에 **"이 일정으로 렌터카 예약"** 버튼
- 날씨 페이지에 **파트너 실내 시설 광고 영역** 만들기

## 3. [그로스] SEO/바이럴 전략의 현실 점검

### 기획서의 가정 vs 현실

| 기획서 가정 | 현실 |
|------------|------|
| "SEO로 월 30,000명 유입" | 네이버 외부 웹사이트 점유율 8%. 월 2,400명이 현실적 |
| "K-Factor 0.4로 바이럴" | 1.0 미만 = 자체 성장 불가. 유료 획득 필수 |
| "제주 카페 6,000개 입점" | 제주 전체 카페 약 3,800개. 100% 이상 불가능 |
| "카페패스 월 15,600건 판매" | MAU 120K 기준 13% 구매율. 현실적으로 3~5% |

### 제안: 현실적 그로스 전략

**Phase 1 — 콘텐츠 SEO (지금 바로)**
- 날씨 지도를 **"제주 오늘 날씨"** 검색 랜딩으로 활용
- 각 지역별 페이지 생성: "애월 날씨", "성산 날씨" → 롱테일 키워드 점령
- 메타 태그, 구조화 데이터, 사이트맵 최적화

**Phase 2 — 네이버 블로그 연동 (2주)**
- AI 코스 메이커 결과를 네이버 블로그 포맷으로 자동 변환
- "제주 2박3일 커플 코스" 같은 블로그 글 자동 생성
- 네이버 검색 22% 커버리지 확보 가능

**Phase 3 — 커뮤니티 (1개월)**
- 코스 공유 기능: 카카오톡/인스타 공유 시 OG 이미지 자동 생성
- "이 코스 따라가기" → 앱 다운로드/회원가입 유도

## 4. [데이터] 데이터 파이프라인 설계가 없다

### 현재
각 프로젝트가 독립적으로 API 호출. 데이터 공유/캐싱/분석 구조 없음.

### 문제
- 날씨 API: 각 프로젝트가 각각 호출하면 할당량 초과 위험
- 관광 데이터: 코스 메이커, 플래너, 지도가 각각 같은 데이터 따로 관리
- 사용자 데이터: 누가 무엇을 검색/생성/예약했는지 추적 불가

### 제안: 공통 데이터 레이어

```
[기상청 API] ─┐
[관광공사 API] ─┤
[에어코리아 API]─┼── [공통 데이터 서버 (Supabase)] ── 모든 프로젝트
[카카오 API] ──┤        ↓
[사용자 행동] ──┘   [분석 대시보드]
```

**구체적으로**:
1. **날씨 캐시 테이블** — 10분마다 갱신, 모든 프로젝트가 공유
2. **POI(장소) 테이블** — 관광공사 데이터 일 1회 동기화
3. **사용자 행동 로그** — 어떤 코스가 인기인지, 어떤 날씨에 검색 폭증하는지
4. **생성된 코스/일정** — AI가 만든 결과를 DB에 저장 → 인기 코스 랭킹

## 5. [프론트] UX 일관성과 온보딩 부재

### 현재
- 9개 앱이 각각 다른 색상 테마 (emerald, violet, sky, indigo...)
- DevNav가 프로젝트 내부 네비게이션만 제공, 프로젝트 간 이동 불가
- 첫 방문자가 "이 서비스가 뭘 하는 건지" 이해하기 어려움

### 제안

**1. 통합 네비게이션**
```
┌─────────────────────────────────────────────────────────┐
│ 🌴 제주패스  [날씨] [지도] [AI 플래너] [코스] [예약] [CS] │
└─────────────────────────────────────────────────────────┘
```
- 모든 프로젝트 상단에 공통 GNB
- 현재 위치 하이라이트
- 프로젝트 간 자유 이동

**2. 포탈 → 여행 가이드로 전환**
현재 포탈은 "프로젝트 목록"인데, 여행자 입장에서는 의미 없음.

```
현재: "Kaflix 아이디어 놀이터" — 9개 프로젝트 카드 나열
개선: "제주 여행, 어디서부터?" — 여행 단계별 가이드
     ① 날씨 확인 → ② 코스 만들기 → ③ 일정 짜기 → ④ 예약하기
```

**3. 디자인 시스템 통일**
- 색상: 제주패스 브랜드 컬러 1개로 통일 (현재 프로젝트별 7종)
- 폰트/간격/버튼: 공통 컴포넌트 라이브러리
- 카드/패널: 일관된 border-radius, shadow, spacing

## 6. 종합 — 가장 임팩트 큰 개선 5가지

| 순위 | 개선 사항 | 효과 | 공수 |
|------|----------|------|------|
| **1** | **프로젝트 간 딥링크 연결** (날씨→코스→플래너→예약) | 전환율 5→15% | 1주 |
| **2** | **모든 AI 결과에 렌터카/예약 CTA** 삽입 | 즉시 매출 발생 | 3일 |
| **3** | **날씨 페이지 SEO 최적화** (지역별 랜딩) | 월 유입 +2,000 | 1주 |
| **4** | **통합 GNB + 포탈 리뉴얼** (여행 가이드 형태) | 이탈률 -30% | 1주 |
| **5** | **공통 데이터 레이어** (날씨 캐시 + POI 테이블) | API 비용 -70% | 2주 |

## 8. 결론

> **현재 기획서는 "무엇을 만들까"에 집중**되어 있음.
> **빠진 것은 "만든 것들을 어떻게 연결해서 돈을 벌까".**

가장 시급한 건 기술 구현이 아니라:
1. 프로젝트를 **하나의 여행 플로우**로 엮는 것
2. 모든 화면에 **예약/결제 CTA**를 넣는 것
3. **실제 사용자 데이터**를 수집해서 가설을 검증하는 것
