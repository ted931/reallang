# The Creative Sandbox — 사내 AI 아이디어 놀이터

> 비개발자 사원들이 Claude와 Supabase를 활용해 자신의 아이디어를 직접 프로토타이핑하고, 안전하게 협업할 수 있는 무한 캔버스 기반의 놀이터

---

## 1. 프로젝트 비전 (Vision)

- **"아이디어에서 실행까지 10분"**: 코딩을 몰라도 Claude와 대화하며 실제 작동하는 UI를 구현
- **"안전한 창의성"**: 개인별 보안 영역과 아이디어 소유권 보호를 통해 안심하고 상상
- **"놀이로서의 개발"**: 무한 캔버스 위에서 자유롭게 낙서하고 조립하는 '개발자 놀이' 환경

## 2. 사용자 페르소나

### A. 마케팅 기획 파트장 (김수진, 34세)
- **상황**: 캠페인 기획은 능숙하지만, 개발팀에 "이런 이벤트 페이지 가능한가요?" 물어보면 2주 뒤에 답이 온다. PPT 와이어프레임을 보내면 "정확히 뭘 원하시는 건지..." 회신.
- **사용 시나리오**: 시즌 프로모션 랜딩 페이지 직접 프로토타이핑, 고객 설문/투표 위젯 제작, 경쟁사 스크린샷 → "우리 브랜드 톤으로 바꿔줘" 요청
- **성공 기준**: "개발팀 대기 없이 마케팅 아이디어를 당일 시각화하고, 팀 내부 피드백까지 받을 수 있다."

### B. B2B 영업 사원 (박준혁, 29세)
- **상황**: 거래처 미팅에서 "이런 대시보드 있으면 좋겠다"는 요청을 자주 듣지만, 엑셀 정리가 한계.
- **사용 시나리오**: 거래처별 매출 현황 대시보드, 견적 계산기 위젯, 영업 파이프라인 시각화
- **성공 기준**: "미팅 전날 밤 30분이면 고객 맞춤형 데모 화면을 뚝딱 만든다."

### C. HR 담당자 (이지은, 31세)
- **상황**: 온보딩 체크리스트를 구글 시트로 관리 중. IT팀에 시스템화 요청하면 우선순위에서 밀린다.
- **사용 시나리오**: 신규 입사자 온보딩 체크리스트 앱 (DB 연동), 사내 만족도 조사 + 실시간 차트, 교육 일정 캘린더
- **성공 기준**: "IT팀 의존 없이 HR 업무에 필요한 소규모 도구를 직접 만들어 쓴다."

### D. 전략기획 팀장 (최상우, 42세)
- **상황**: 경영진 보고 자료를 만들지만, 정적 PPT 차트의 한계를 느낀다. BI 도구 도입은 예산 승인 필요.
- **사용 시나리오**: 인터랙티브 KPI 대시보드, 재무 시뮬레이터 위젯, 부서별 성과 비교 차트
- **성공 기준**: "경영진이 '이거 어떻게 만들었어?'라고 물어보는 수준의 보고서를 혼자 만든다."

## 3. 핵심 기능 (Core Features)

### 3-1. 무한 캔버스 (Infinite Workspace)
- **자유로운 발상**: Apple 프리폼/FigJam처럼 공간 제약 없이 텍스트, 이미지, 드로잉 배치
- **AI 위젯 박제**: Claude가 생성한 React 코드를 캔버스 위에 붙여넣으면 **'살아있는 앱'**으로 즉시 실행
- **팀 협업**: 공유 구역에서 실시간 입장 + 포스트잇 피드백

### 3-2. 개인별 대시보드 및 보안 (Privacy & Security)
- **독립 공간**: Supabase Auth로 각 사원에게 고유 개인 대시보드 할당
- **잠금 기능**: 아이디어는 기본 '나만 보기'. 비밀번호 잠금, 특정 팀원만 초대 가능
- **소유권 보호**: 최초 생성 시점을 불변 타임스탬프로 기록 → 아이디어 도용 방지, 최초 제안자 증명

### 3-3. 비개발자 친화적 DB 연동 (No-Code Data)
- **치트키 함수**: 관리자가 설정한 Supabase 테이블을 `saveData("orders", {...})` 한 줄로 연동
- **간편 템플릿**: Claude에게 "DB 연동해서 저장 기능 만들어줘"라고 요청하면 즉시 코드 생성

### 4-2. 핵심 설계 결정: tldraw Custom Shape

캔버스 내부에서 React 컴포넌트를 실행하는 3가지 방식을 비교:

| 방식 | 장점 | 단점 | 판정 |
|------|------|------|------|
| **tldraw Custom Shape** | 캔버스 좌표계에 네이티브 통합, 드래그/리사이즈/줌 자동 | tldraw 종속 | **채택** |
| iframe 오버레이 | 완전 격리 | 줌/팬 동기화 극도로 복잡, 수십 개 시 성능 붕괴 | 탈락 |
| Excalidraw 커스텀 | 오픈소스 | React 컴포넌트 주입 공식 경로 부재, API 미성숙 | 탈락 |

tldraw `ShapeUtil.component()`에서 React JSX를 네이티브로 반환할 수 있어 캔버스 좌표계에 완전 통합된다.

| 영역 | 선택 | 근거 |

## 5. Supabase 보안 아키텍처

### 5-1. 테이블 구조

```sql
-- 핵심 테이블
ideas (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  owner_id uuid REFERENCES auth.users(id) NOT NULL,
  title text NOT NULL,
  canvas_state jsonb,
  visibility text DEFAULT 'private' CHECK (visibility IN ('private','team','public')),
  password_hash text,          -- 비밀번호 보호 구역 (bcrypt)
  created_at timestamptz DEFAULT now(),
  updated_at timestamptz DEFAULT now()
);

-- 팀 공유 권한
idea_permissions (
  idea_id uuid REFERENCES ideas(id) ON DELETE CASCADE,
  user_id uuid REFERENCES auth.users(id),
  role text CHECK (role IN ('viewer','editor','admin')),
  granted_by uuid REFERENCES auth.users(id),
  granted_at timestamptz DEFAULT now(),
  PRIMARY KEY (idea_id, user_id)
);

-- 변조 방지 감사 로그 (APPEND-ONLY)
idea_audit_log (
  id bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  idea_id uuid REFERENCES ideas(id),
  actor_id uuid REFERENCES auth.users(id),
  action text NOT NULL,           -- 'created','edited','shared','locked'
  snapshot_hash text NOT NULL,    -- SHA-256(canvas_state)
  created_at timestamptz DEFAULT now()
);
```

### 5-2. RLS 정책

```sql
-- 1. 소유자 본인만 접근
CREATE POLICY "owner_access" ON ideas
  FOR ALL USING (owner_id = auth.uid());

-- 2. 팀 공유: 권한 부여된 사용자 조회
CREATE POLICY "shared_access" ON ideas
  FOR SELECT USING (
    visibility = 'team' AND EXISTS (
      SELECT 1 FROM idea_permissions
      WHERE idea_id = ideas.id AND user_id = auth.uid()
    )
  );

-- 3. 공개 아이디어: 전체 조회
CREATE POLICY "public_read" ON ideas
  FOR SELECT USING (visibility = 'public');

-- 4. 편집: editor/admin만 수정
CREATE POLICY "edit_access" ON ideas
  FOR UPDATE USING (
    owner_id = auth.uid() OR EXISTS (
      SELECT 1 FROM idea_permissions
      WHERE idea_id = ideas.id AND user_id = auth.uid()
        AND role IN ('editor','admin')
    )
  );

-- 5. 감사 로그: INSERT만 허용 (변조 방지)
CREATE POLICY "audit_append_only" ON idea_audit_log
  FOR INSERT WITH CHECK (actor_id = auth.uid());

-- created_at 불변 보장 트리거
CREATE OR REPLACE FUNCTION prevent_created_at_update()
RETURNS TRIGGER AS $$
BEGIN
  NEW.created_at := OLD.created_at;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER immutable_created_at
  BEFORE UPDATE ON ideas
  FOR EACH ROW EXECUTE FUNCTION prevent_created_at_update();
```

**비밀번호 보호**: 클라이언트에서 비밀번호 입력 → API Route에서 bcrypt 검증 → 단기 JWT 클레임(`idea_access`) 발급 → RLS에서 `auth.jwt()->'idea_access'`로 검증.

**소유권 타임스탬핑**: `idea_audit_log`의 최초 `created` 레코드가 법적 증거력. `snapshot_hash`(SHA-256)로 사후 변조 탐지. DELETE/UPDATE RLS 정책 부재로 기록 수정/삭제 완전 차단.

## 6. 캔버스 내 코드 실행 아키텍처

### 6-1. tldraw Custom Shape: LiveWidgetShape

```tsx
class LiveWidgetShapeUtil extends ShapeUtil<LiveWidgetShape> {
  static type = 'live-widget' as const

  getDefaultProps() {
    return { code: '<div>Hello World</div>', w: 320, h: 240 }
  }

  component(shape: LiveWidgetShape) {
    return (
      <HTMLContainer>
        <div
          style={{ width: shape.props.w, height: shape.props.h }}
          onPointerDown={(e) => e.stopPropagation()}  // 위젯 내부 클릭 허용
        >
          <ErrorBoundary fallback={<CrashCard shapeId={shape.id} />}>
            <LiveProvider code={shape.props.code} scope={sandboxScope}>
              <LivePreview />
              <LiveError className="text-red-500 text-xs p-2" />
            </LiveProvider>
          </ErrorBoundary>
        </div>
      </HTMLContainer>
    )
  }
}
```

### 6-2. 격리 전략
- react-live `scope`에 허용 라이브러리만 주입 (React, 치트키 API, 회사 컴포넌트)
- `import`, `require`, `fetch`, `eval`, `document`, `window` 키워드를 정적 분석으로 차단

### 6-3. 성능 (다수 위젯)
- tldraw 뷰포트 컬링: 화면 밖 Shape 자동 언마운트 → 100개 위젯이어도 보이는 5~10개만 렌더
- `React.memo`로 코드 미변경 Shape의 리렌더 방지

### 6-4. 에러 경계
- 모든 `LiveProvider`는 개별 `ErrorBoundary`로 감쌈
- 하나의 위젯 크래시 → 해당 Shape만 에러 표시 ("코드에 오류가 있습니다. Claude에게 수정을 요청하세요")

### 6-5. 스케치 → 위젯 파이프라인

```tsx
async function sketchToWidget(editor: Editor, selectedArea: Box2d) {
  // 1. 선택 영역을 이미지로 캡처
  const svg = await editor.getSvg([...editor.getSelectedShapeIds()])
  const imageBase64 = await svgToBase64(svg)

  // 2. Claude Vision → 코드 생성 (SSE 스트리밍)
  const stream = await generateWidgetCode(imageBase64, '이 스케치를 React UI로 구현해줘')
  let code = ''
  for await (const chunk of stream) { code += chunk }

  // 3. LiveWidgetShape로 캔버스에 삽입
  editor.createShape({
    type: 'live-widget',
    x: selectedArea.x + selectedArea.w + 20,
    y: selectedArea.y,
    props: { code, w: 400, h: 300 },
  })
}
```

## 7. "치트키" API 설계

### 7-1. TypeScript 인터페이스

```typescript
interface CheatKeyAPI {
  saveData: (table: string, data: Record<string, any>) => Promise<{ id: string }>;
  loadData: (table: string, filter?: Record<string, any>) => Promise<any[]>;
  loadOne: (table: string, id: string) => Promise<any | null>;
  updateData: (table: string, id: string, changes: Record<string, any>) => Promise<void>;
  deleteData: (table: string, id: string) => Promise<void>;
  me: () => { id: string; name: string; email: string };
}
```

### 7-2. 사용 예시 (비개발자가 Claude에게 요청하여 받을 코드)

```tsx
function CafeOrder() {
  const [menu, setMenu] = React.useState([]);
  const user = me();

  React.useEffect(() => {
    loadData('cafe_menu').then(setMenu);
  }, []);

  const handleOrder = async (item) => {
    await saveData('orders', {
      item_name: item.name,
      ordered_by: user.name,
      status: '접수됨'
    });
    alert(`${item.name} 주문 완료!`);
  };

  return (
    <div style={{ padding: 16 }}>
      <h2>{user.name}님, 오늘 뭐 드실래요?</h2>
      {menu.map(item => (
        <button key={item.id} onClick={() => handleOrder(item)}>
          {item.name} - {item.price}원
        </button>
      ))}
    </div>
  );
}
```

### 7-3. 내부 구현

```typescript
export const cheatkeyScope = {
  saveData: async (tableName: string, data: Record<string, unknown>) => {
    const { data: result, error } = await supabase
      .from(`sandbox_${tableName}`)  // 접두사로 격리
      .insert({ ...data, _created_by: (await supabase.auth.getUser()).data.user?.id })
      .select().single()
    if (error) throw new Error(`저장 실패: ${error.message}`)
    return result
  },

  loadData: async (tableName: string, filter?: Record<string, unknown>) => {
    let query = supabase.from(`sandbox_${tableName}`).select('*')
    if (filter) {
      Object.entries(filter).forEach(([key, value]) => {
        query = query.eq(key, value)
      })
    }
    const { data, error } = await query.order('created_at', { ascending: false })
    if (error) throw new Error(`조회 실패: ${error.message}`)
    return data
  },

  deleteData: async (tableName: string, id: string) => {
    const { error } = await supabase
      .from(`sandbox_${tableName}`)
      .delete()
      .eq('id', id)
      .eq('_created_by', (await supabase.auth.getUser()).data.user?.id) // 본인만
    if (error) throw new Error(`삭제 실패: ${error.message}`)
    return true
  },
}
```

**보안**: 모든 치트키 함수는 `/api/cheatkey/[table]` 경유. 허용 테이블 화이트리스트(`cafe_menu`, `orders`, `feedback` 등)에 대해서만 쿼리 실행. 미등록 테이블 접근 시 403 차단.

## 8. 사용자 여정 맵 (UX Journey)

### 시나리오 플로우
1. **아이디어 발산**: 사원이 캔버스에 "사내 카페 자동 주문 앱" 아이디어를 낙서
2. **Claude 협업**: Claude에게 낙서를 보여주며 "이걸 토스 앱 느낌의 UI 코드로 짜줘" 요청
3. **코드 적용**: 생성된 코드가 캔버스 위에 '살아있는 앱'으로 렌더링
4. **데이터 테스트**: `saveData` 치트키로 실제 DB에 주문 데이터 저장 확인
5. **보안 및 공유**: "비밀번호 설정"으로 보호 → 준비되면 "팀 공유"로 전환 → 피드백 수집

## 9. 온보딩 전략: 두려움을 호기심으로

### 원칙: "실패할 수 없는 첫 경험"

### 단계 1: 제로 마찰 진입 (0~30초)
- 로그인 직후 빈 캔버스가 아닌 **"오늘의 영감"** 화면
- 다른 사원들이 만든 인기 위젯 3개가 실제 작동 상태로 전시
- 큰 버튼: **"나도 만들어 보기"**

### 단계 2: "나의 첫 AI 앱 만들기" 5분 튜토리얼
| 분 | 활동 | 달성감 |
|----|------|--------|
| 1분 | 캔버스에 텍스트 적기 → 포스트잇 스타일로 자동 변환 | 캔버스 친숙해지기 |
| 2분 | AI에게 "날씨 위젯 만들어줘" → 살아있는 위젯 등장 | "와, 이게 되네?" 모먼트 |
| 3분 | "배경을 파란색으로 바꿔줘" → 대화로 수정 체험 | 코드 없이 수정 가능 체득 |
| 4분 | 버튼 클릭 → saveData로 실제 DB 저장 경험 | 데이터 연결의 힘 |
| 5분 | 완성된 위젯을 갤러리에 공유 → 튜토리얼 완료 배지 | 성취감 + 사회적 공유 |

### 단계 3: 템플릿 갤러리 (Progressive Disclosure)
- **초급** (별 1개): 팀 투표, 점심 추천기, D-day 카운터
- **중급** (별 2개): 칸반 보드, 설문조사+차트, 계산기
- **고급** (별 3개): 매출 대시보드, 재고 관리, 피드백 분석기

### 단계 4: 점진적 기능 해금
- 첫 위젯: 기본 도구 + AI 채팅만
- 3개 후: 고급 레이아웃, 커스텀 스타일링
- 5개 후: DB 직접 연동, API 호출, 복합 위젯
- 10개 후: 코드 직접 편집, 커스텀 컴포넌트 등록

## 11. UX 차별화: ChatGPT/Claude 직접 사용과 뭐가 다른가

### 공간적 사고 (Spatial Thinking)
ChatGPT는 선형 대화. 아이디어는 비선형. 캔버스에서 고객 설문 위젯(왼쪽) + 데이터 차트(가운데) + 랜딩페이지(오른쪽)를 나란히 놓고 한눈에 볼 수 있다.

### 영속적 결과물 (Persistent Artifacts)
ChatGPT 코드는 대화창 닫히면 사라진다. Sandbox 위젯은 대시보드에 영구 저장, URL 하나로 공유, 실제 작동 상태 유지. "일회용 답변"이 아니라 **"사내 자산"**.

### 회사 맥락 내장 AI (Company Context)
- 우리 회사 브랜드 컬러와 디자인 시스템 자동 적용
- 사내 DB 테이블 구조를 이미 알고 있어 데이터 연동 즉각적
- 조직도, 공용 API, 기존 도구와의 연결 사전 설정

### 사회적 증거와 협업 (Social Proof)
"영업팀 박준혁 씨가 이런 걸 만들었네? 나도 비슷한 거 만들어볼까?" — 자연스러운 동기 부여.
