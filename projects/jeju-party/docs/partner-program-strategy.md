# 파트너 광고/할인 프로그램 — 독립 사이트 vs 애드온 메뉴 전략 분석

> 작성일: 2026-04-28  
> 분석 대상: jeju-party (포트 3010) 파트너 광고/할인 프로그램  
> 관련 서비스: jeju-biz (포트 3022), jeju-car (포트 3021), monorepo 전체 16개 프로젝트

---

## 0. 현황 파악 — 코드베이스에서 확인된 사실

코드베이스를 직접 읽은 결과, 파트너 기능은 이미 다음 수준까지 구현되어 있다.

| 구성요소 | 파일 경로 | 현황 |
|---|---|---|
| `PartnerOffer` 타입 | `src/lib/types.ts` | `sponsorLevel: "basic" \| "featured"` 2단계 이미 정의 |
| 더미 파트너 10개 | `src/lib/dummy-partners.ts` | 게스트하우스·카페·렌탈·식당·관광지·액티비티 전 업종 포함 |
| `FeaturedPartnerCard` | `src/components/partner-offer-card.tsx` | 풀 width 주황 테두리 카드, 코드 복사 기능 |
| `BasicPartnerCard` | `src/components/partner-offer-card.tsx` | 가로 스크롤 컴팩트 카드 |
| 파티 상세 페이지 연동 | `src/app/party/[id]/page.tsx` | 카테고리 매칭 후 인라인 노출 이미 구현 |
| B2B 대시보드 셸 | `projects/jeju-biz/` (포트 3022) | 예약·CS·통계 대시보드가 별도 앱으로 존재 |

**결론적으로, 사용자 facing(C2C)은 jeju-party에 구현됐고, 업체 facing(B2B) 셸은 jeju-biz에 이미 존재한다.** 따라서 이 결정은 "만들 것인가"가 아니라 "어디에 붙일 것인가"의 문제다.

---

## A. 독립 사이트 분리

### 장점

1. **브랜드 명확성** — 업체 담당자가 `partner.jejupass.com`이나 `/biz`에 접근하면 "나만을 위한 도구"라는 신뢰감을 준다. 프리프(FRIP) 파트너 센터, 당근비즈(당근마켓 사장님 앱)가 대표적 성공 사례.
2. **SEO/마케팅 분리** — "제주 관광지 광고 플랫폼" 키워드를 별도 도메인으로 공략 가능.
3. **기존 서비스 UX 오염 없음** — 여행자용 앱과 업체 관리 화면이 섞이지 않는다.
4. **확장성** — jeju-car, jeju-travel 등 다른 플랫폼의 파트너 계정을 통합 관리하는 단일 B2B 포털로 성장 가능. 이미 jeju-biz(3022)가 이 역할을 준비 중.

### 단점

1. **초기 트래픽 제로** — 독립 사이트를 업체에 설득하려면 "우리 플랫폼의 MAU/DAU"를 증명해야 한다. 지금은 그 숫자가 없다.
2. **온보딩 비용** — 업체가 새 사이트에 가입·로그인하는 마찰이 크다. B2B SaaS 온보딩 완주율은 일반적으로 40% 이하.
3. **데이터 연동 복잡도** — 파티의 카테고리·지역·일정과 파트너 오퍼를 매칭하는 로직이 API 경계를 넘어야 한다. 현재 `PARTNER_OFFERS`는 `targetCategories: string[]`로 단순 배열이지만, 실 운영에서는 실시간 동기화가 필요하다.
4. **개발/운영 비용** — PM2 프로세스를 하나 더 관리해야 하고, Nginx 라우팅도 추가된다.

---

## B. 기존 서비스 내 애드온 메뉴

### 장점

1. **기존 트래픽 활용** — 파티 상세 페이지(`/party/[id]`)는 이미 사용자가 가장 오래 머무는 화면이다. `FeaturedPartnerCard`가 이미 이 페이지에 인라인 노출 중.
2. **파티 데이터 즉시 연동** — `party.category`와 `offer.targetCategories` 매칭이 이미 클라이언트 사이드에서 작동한다. API 경계 없음.
3. **낮은 초기 개발 비용** — 업체 관리 화면을 `/create` 플로우 뒤에 `/partner/register` 라우트 하나로 붙이면 된다.
4. **검증 후 스핀오프 가능** — 트래픽과 전환율 데이터가 쌓이면 jeju-biz 쪽으로 이전하는 판단을 데이터 기반으로 할 수 있다.

### 단점

1. **업체 관리 UI 혼재** — 여행자 앱에 업체 로그인 영역이 생기면 IA(정보 구조)가 복잡해진다.
2. **광고 UX 오염** — 파티 상세 페이지에 광고 카드가 과도해지면 "참가 CTA"의 시선을 빼앗는다. 현재 구현은 Featured 1개 + Basic 가로 스크롤로 잘 분리되어 있으나, 업체 수가 늘면 취약해진다.
3. **브랜드 포지셔닝 혼란** — "여행자 파티 앱"이 동시에 "업체 광고 플랫폼"처럼 느껴지면 초기 여행자 사용자의 이탈 가능성이 있다.

## C. 하이브리드 — 권장 전략

**핵심 원칙: 사용자(C2C) 화면은 jeju-party 내에, 업체(B2B) 화면은 jeju-biz에.**

이 구조는 이미 monorepo 아키텍처에 자연스럽게 담겨 있다. 추가 인프라 없이 경계만 명확히 정의하면 된다.

```
[여행자 경험]                    [업체 경험]
jeju-party (3010)                jeju-biz (3022)
  /party/[id]                      /partner/dashboard
    └─ FeaturedPartnerCard           └─ 통계 (노출수·클릭수·쿠폰 사용)
    └─ BasicPartnerCard (스크롤)     └─ 오퍼 등록/수정
                                     └─ 결제 (월정액·CPA 정산)
                                     └─ 파티 카테고리 타겟팅 설정
```

### 데이터 흐름

```
jeju-biz (업체가 오퍼 등록)
    ↓ API 또는 공유 Supabase 테이블
jeju-party (카테고리 매칭 후 노출)
    ↓ 클릭/쿠폰 사용 이벤트
jeju-biz (통계 대시보드 업데이트)
```

현재 `PARTNER_OFFERS`는 `dummy-partners.ts`에 정적 배열로 존재한다. Supabase로 전환 시 (`src/lib/supabase.ts`가 이미 준비됨) 양쪽 앱이 같은 테이블을 바라보면 된다.

## 유사 서비스 사례 분석

| 서비스 | 사용자 facing | 업체 facing | 시사점 |
|---|---|---|---|
| **프리프(FRIP)** | 액티비티 피드에 인라인 파트너 카드 | `partner.frip.in` 별도 B2B 포털 | MVP → 별도 포털 전환 순서로 성장. 초기에는 인라인 노출만 운영 |
| **당근마켓** | 게시글 사이에 비즈프로필 카드 | `biz.daangn.com` 완전 분리 | 트래픽 확보 후 B2B 포털 런칭. 분리 타이밍: MAU 500만 이후 |
| **소모임** | 모임 상세에 협찬 배너 | 내부 영업팀 직접 관리(대시보드 없음) | 스타트업 초기에는 셀프서브 대시보드 없이 영업팀이 수동 운영 |
| **야놀자 파트너** | 상품 리스트에 광고 슬롯 | `partner.yanolja.com` | 수익화 규모가 커진 후 전용 포털 투자. 초기 인라인 광고만으로 BEP 달성 |

**공통 패턴**: 모든 서비스가 "사용자 앱 내 인라인 노출 → 트래픽 증명 → B2B 포털 분리" 순서를 따랐다. 처음부터 독립 사이트를 만든 경우는 없다.

---

## MVP 단계 판단 기준

### 1. 트래픽이 없으면 B2B 포털은 아무 의미가 없다

업체가 파트너 대시보드에 가입하는 이유는 단 하나: **노출 효과**다. 노출 효과를 증명하려면 먼저 여행자 트래픽이 있어야 한다. 독립 사이트를 먼저 만드는 것은 방문자 없는 광고판을 파는 것과 같다.

### 2. 업체 담당자의 첫 번째 질문은 "이 앱 사용자가 몇 명이에요?"다

파트너사 영업 미팅에서 가장 먼저 나오는 질문이다. jeju-party의 파티 참가자 수치 없이 독립 B2B 사이트를 론칭하면 설득력이 없다.

### 3. CPA 수익 모델은 쿠폰 사용 추적이 전제다

현재 `discountCode`는 문자열로만 존재한다. CPA 모델을 실제로 운영하려면 코드 발급·추적·정산 시스템이 필요한데, 이는 B2B 포털의 핵심 기능이다. 이 기능은 여행자 트래픽이 일정 수준에 도달했을 때 개발해도 늦지 않다.

## 단계별 실행 로드맵

### Phase 0 — 현황 정비 (1~2주)

**목표**: 이미 구현된 파트너 카드를 안정화하고, 수동 계약 기반으로 첫 유료 파트너 유치

- [ ] `dummy-partners.ts`를 Supabase `partner_offers` 테이블로 마이그레이션 (`src/lib/supabase.ts` 활용)
- [ ] 파티 상세 페이지의 파트너 카드 노출 순서 정렬 로직 확정 (featured 우선, 카테고리 매칭 score 기준)
- [ ] 쿠폰 클릭 이벤트 로깅 추가 (GA4 또는 Supabase 이벤트 테이블)
- [ ] 첫 3개 파트너사 수동 계약 (구두 계약 + 스프레드시트 관리로 시작)
- [ ] 파트너 노출 영역에 "제휴 문의" 링크 추가 → 구글폼 또는 이메일 수집

**수익 모델**: 정액 게재비 (월 5~10만원) 구두 계약만으로 충분.

---

### Phase 1 — 인라인 최적화 (1개월)

**목표**: 여행자 UX를 해치지 않으면서 파트너 오퍼 노출 효율 극대화

- [ ] `FeaturedPartnerCard`를 파티 참가 완료 화면(`paymentStep === "done"`) 직후에 노출 — 구매 직후가 가장 전환율 높은 시점
- [ ] 파티 목록 피드에 "파트너 혜택 있는 파티" 뱃지 추가 (카드에 작은 아이콘 표시)
- [ ] 쿠폰 복사 후 "파트너 사이트 바로가기" CTA 추가
- [ ] A/B 테스트: Featured 카드 위치 (파티 상세 상단 vs 참가 버튼 하단)
- [ ] 주간 노출·클릭·쿠폰 복사 수치를 수동으로 파트너사에 리포팅

**KPI 목표**: 쿠폰 복사율 15% 이상, 파트너사 재계약률 80% 이상

---

### Phase 2 — 셀프서브 B2B 미니 대시보드 (2~3개월)

**목표**: 파트너사가 스스로 오퍼를 등록·수정·통계 확인할 수 있는 최소 화면 구축

**구현 위치**: `projects/jeju-biz/src/app/partner/` 신규 라우트 (jeju-biz 3022 내)

```
/partner/register      — 업체 등록 (사업자명, 업종, 위치, 담당자)
/partner/offers/new    — 오퍼 등록 (오퍼 타입, 할인율, 타겟 카테고리, 유효기간)
/partner/dashboard     — 통계 (노출수·클릭수·쿠폰 복사수·추정 방문자)
/partner/billing       — 정산 (정액제 청구서, CPA 정산 내역)
```

jeju-biz의 기존 대시보드 컴포넌트(`src/components/dev-feature-panel.tsx`, 통계 바 차트 등)를 재활용하면 개발 비용 최소화.

**수익 모델 고도화**:
- 정액 게재비: 셀프서브 결제 (토스페이먼츠 위젯)
- CPA 쿠폰 수수료: 쿠폰 복사 이벤트 → Supabase 기록 → 월말 정산
- 경로 스폰서십: jeju-car의 렌터카 예약 완료 화면에 파트너 숙소 노출 (jeju-car ↔ jeju-biz API 연동)

---

### Phase 3 — 멀티 플랫폼 B2B 포털 (6개월+)

**목표**: jeju-party 외 jeju-car, jeju-travel 등 전 플랫폼 파트너를 통합 관리하는 독립 포털

**조건**: Phase 2에서 파트너사 30개 이상, 월 정산액 300만원 이상 달성 후 진행

- jeju-biz를 완전한 파트너 포털로 확장
- 별도 서브도메인 `biz.{도메인}` 브랜딩
- 파트너 등급 시스템 (Starter → Pro → Enterprise)
- jeju-car 연동: 렌터카 예약 완료 후 파트너 숙소/액티비티 크로스셀 노출
- 광고 입찰 시스템 (카테고리별 노출 경쟁)

## 핵심 판단 기준 3가지

### 1. 트래픽이 먼저다 — B2B 포털은 트래픽의 결과물이지 원인이 아니다

독립 B2B 사이트를 만들면 업체 영업 미팅에서 "우리 플랫폼의 트래픽"을 물어볼 때 대답할 숫자가 없다. jeju-party의 파티 참가자 수가 월 1,000명을 넘기 전에는 B2B 포털에 투자하는 것이 선행 조건 역전이다.

### 2. 여행자 UX가 파트너 신뢰의 원천이다

파트너사가 광고비를 지불하는 이유는 "여행자가 이 앱을 신뢰하고 쿠폰을 쓰기 때문"이다. 광고를 너무 많이 넣어 여행자 UX를 망치면 파트너 프로그램 전체가 무너진다. 현재 `FeaturedPartnerCard`(1개) + `BasicPartnerCard`(가로 스크롤) 구조는 이 균형을 잘 잡고 있다. 이 구조를 지키는 것이 곧 파트너 가치를 지키는 것이다.

### 3. jeju-biz는 이미 있다 — 새로 만들지 말고 연결하라

`projects/jeju-biz`(포트 3022)는 예약·CS·통계 대시보드를 갖춘 B2B 앱이 이미 존재한다. 파트너 관리 화면을 이 앱의 `/partner` 라우트로 추가하는 것이 새 독립 사이트를 만드는 것보다 PM2 프로세스·Nginx·도메인·유지보수 비용을 절감하면서도 동일한 효과를 낸다.

## 최종 권장 전략 요약

```
지금 당장 → Phase 0: Supabase 연동 + 수동 계약 3개 파트너
1개월 후  → Phase 1: 참가 완료 화면 최적화 + KPI 측정
3개월 후  → Phase 2: jeju-biz /partner 라우트로 셀프서브 B2B 미니 대시보드
6개월 후  → Phase 3: 트래픽·수익 KPI 달성 시 완전 독립 포털 전환
```

**독립 사이트 분리는 Phase 3의 선택지다. MVP 단계에서 독립 사이트를 먼저 만드는 것은 증명되지 않은 트래픽에 B2B 비용을 쓰는 것이고, 이는 스타트업에서 가장 흔한 실패 패턴 중 하나다.**
